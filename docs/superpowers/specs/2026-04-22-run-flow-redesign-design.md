# 任务处理流重构设计

**目标**

把当前“任务创建 → 后处理 → 结果展示 → 历史记录”的流程，重构为更符合实际使用顺序的四页流程：

- `01 任务创建`
- `02 结果展示与后处理`
- `03 统计分析`
- `04 历史记录`

重构后的核心原则是：

1. 任务创建页负责定义这次任务怎么跑，并负责启动主分割
2. 用户先看主分割结果，再决定是否做后处理
3. 统计页只针对“已确认结果”做统计，不对未确认预览结果做统计

**现状问题**

当前流程把“后处理配置”放在主分割结果出现之前，用户必须在没看到实际分割效果时就先决定是否填孔、watershed、形状过滤和触边剔除。这会导致：

- 后处理配置时机过早，认知顺序不自然
- 用户无法基于实际结果判断后处理是否必要
- 统计与结果展示职责纠缠在一起，页面定位不清晰

**页面职责**

`01 任务创建`

- 图像导入
- 逐图标定
- 预处理配置
- 分割模式选择
- 草稿任务创建与图像上传
- 主分割启动按钮
- 紧凑的运行进度展示

这一页负责“定义并启动任务”。这里不再放后处理配置，也不展示统计结果。

`02 结果展示与后处理`

- 展示当前已确认结果的原图、掩码、叠加图
- 提供图像切换和结果视图切换
- 提供轻量后处理参数面板
- 提供“应用后处理”动作
- 弹出前后对比确认弹窗

这一页负责“看分割结果，再决定是否修正”。后处理不会直接覆盖当前结果，而是先生成预览结果，用户确认后才替换主界面展示结果。

`03 统计分析`

- 展示当前已确认结果的统计卡片、图表、明细表
- 提供统计导出
- 显示统计口径说明

这一页只做统计，不承担后处理判断职责。用户应在 `02` 页确认结果后，再进入统计页。

`04 历史记录`

- 保持现有职责不变
- 浏览历史任务、回看结果、进入详情

**状态流**

任务运行状态拆成两层：

1. 运行状态
- `draft`
- `queued`
- `running`
- `completed`
- `failed`
- `partial_success`

2. 结果状态
- `base_result`：主分割完成后的基线结果
- `preview_result`：后处理预览结果，只用于确认弹窗
- `confirmed_result`：用户确认后当前界面展示和统计使用的结果

规则如下：

- 主分割完成后，`base_result` 自动成为首个 `confirmed_result`
- 用户在 `02` 页点击“应用后处理”时，后端基于当前 `confirmed_result` 生成 `preview_result`
- 用户取消时，丢弃 `preview_result`
- 用户确认时，`preview_result` 提升为新的 `confirmed_result`
- `03` 页统计只读取 `confirmed_result`

**02 页后处理交互**

`02` 页的后处理动作分两段：

1. 用户调整后处理参数
- 填孔
- watershed
- 平滑
- 形状过滤
- 触边剔除

2. 用户点击“应用后处理”
- 前端请求后端生成一份预览结果
- 弹窗展示前后对比图
- 弹窗底部仅保留：
  - `取消`
  - `确认应用`

弹窗展示规则：

- 默认展示当前选中图像的前后对比
- 弹窗说明文字明确写出：当前展示的是选中图像的前后对比，确认后将对当前结果统一应用后处理

用户确认后：

- 主界面的分割图、叠加图、结果说明刷新为新的 `confirmed_result`
- `03` 页统计标记为“基于最新确认结果可生成”

**统计页规则**

`03` 页只统计“已确认结果”，不统计预览结果。

进入 `03` 页时：

- 如果当前任务存在 `confirmed_result`，直接读取该版本的统计
- 如果 `02` 页刚确认过新的后处理结果，重新生成或重新拉取该结果版本的统计

这样可以确保：

- 用户先做结果判断，再看统计
- 统计永远和当前确认结果一致
- 前端不需要自己修改统计值，只需要切换结果版本并重新拉数据

**后端接口方向**

需要补一条“后处理预览/确认”链路：

1. 主分割结果接口
- 返回 `base_result`

2. 后处理预览接口
- 输入：当前 `confirmed_result` + 后处理参数
- 输出：`preview_result`

3. 后处理确认接口
- 输入：当前 `preview_result`
- 输出：新的 `confirmed_result`

4. 统计接口
- 输入：结果版本标识
- 输出：对应版本的统计结果

后端不要求前端本地拼装新结果。前端只负责提交参数、拉取结果、切换当前版本。

**页面跳转**

顶部导航改为：

- `01 任务创建`
- `02 结果展示与后处理`
- `03 统计分析`
- `04 历史记录`

跳转规则：

- `01` 完成草稿准备后允许启动主分割
- 主分割完成后允许进入 `02`
- `02` 存在已确认结果后允许进入 `03`
- `03` 不再反向承担后处理决策

**错误处理**

- 主分割失败：停留在 `01`，展示运行错误
- 后处理预览失败：停留在 `02`，不覆盖当前结果，只提示预览生成失败
- 后处理确认失败：保持当前 `confirmed_result` 不变
- 统计生成失败：停留在 `03`，提示统计失败，但不影响 `02` 页结果展示

**测试重点**

1. `01` 页启动主分割后，进度展示能随运行状态刷新
2. `02` 页默认展示主分割基线结果
3. `02` 页点击“应用后处理”后，弹窗能展示前后对比
4. 取消预览时，主界面结果不变
5. 确认应用后，主界面刷新为后处理结果
6. `03` 页统计只读取最新 `confirmed_result`
7. 再次后处理并确认后，`03` 页统计能跟随最新结果刷新

**范围外**

本次设计不包含：

- 多历史版本完整管理界面
- 后处理版本回滚列表
- 预览弹窗中的统计差异对比
- 结果批注或人工编辑功能
